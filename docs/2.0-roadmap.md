# desktop_updater 2.0 Roadmap

## Current State

desktop_updater 1.x is the stable folder-based updater line. desktop_updater 2.x is the active development line for a zip-first release format.

## Why Zip-First

Zip-first releases avoid public folder listing, work with signed URLs and private buckets, and make each update artifact easy to verify before extraction.

## Release Contract

The app checks `app-archive.json`, the selected item points to `release.json`, and `release.json` points to one zip artifact with length, SHA-256, and production authenticity metadata.

## Platform Behavior

- macOS replaces a complete `.app` bundle created and extracted with `/usr/bin/ditto`.
- Windows replaces a complete app directory after the running process exits.
- Linux uses one documented strategy for 2.0 final: directory bundle zip unless AppImage support is explicitly selected before release.

## macOS Package Manager Compatibility

Swift Package Manager is the primary macOS plugin integration path for 2.0. The package still keeps the CocoaPods podspec as the fallback compatibility path for apps that explicitly disable SwiftPM or run older Flutter tooling. SwiftPM checks must pass without CocoaPods; CocoaPods checks are a separate fallback gate.

## Private Bucket Guidance

Do not rely on bucket listing. Publish exact descriptor and artifact URLs. Signed URLs, private proxies, and CDN URLs are supported as long as clients can fetch the exact URL before expiry.

## Release Gates

2.0 final requires unit, widget, native, and e2e tests for check, download, verify, stage, install scheduling, rollback, and migration examples.

macOS integration can be run locally on a macOS host:

- SwiftPM-first: enable SwiftPM and run the example app/plugin tests without requiring `pod install`.
- CocoaPods fallback: disable SwiftPM, then build or test the example app to preserve compatibility.
- Production acceptance: build an exported Release `.app`, sign it with a `Developer ID Application` identity from the same Team ID as the installed app, enable hardened runtime, notarize it, staple the ticket, package it with `ditto`, and verify the update through the native production gates. Debug smoke tests may bypass signing only in `#if DEBUG`; Release builds must require bundle ID, Team ID, `codesign`, Gatekeeper, and stapler validation.
- App Sandbox must stay disabled for the whole-app replacement strategy, and production entitlements must not include `get-task-allow`. A sandboxed Mac App Store style app should use the store update channel instead of this self-updater.

Windows and Linux integration are covered by hosted GitHub Actions runners.

## Compatibility

1.x installed apps should remain on the 1.x release contract until their publishers migrate their release pipeline and app code.
