# Migrating From 1.x To 2.0

desktop_updater 2.0 replaces folder-based update directories with a zip-first release contract. The app still starts from `app-archive.json`, but each matching item points to a `release.json` descriptor, and the descriptor points to exactly one verified zip artifact.

## Before: 1.x Folder Contract

```json
{
  "appName": "Example App",
  "items": [
    {
      "version": "1.4.0",
      "shortVersion": 10400,
      "platform": "macos",
      "mandatory": false,
      "url": "https://updates.example.com/example/10400/macos/"
    }
  ]
}
```

The 1.x client expects public or fetchable files below `url`, such as `hashes.json`, payload files, or macOS release manifests.

## After: 2.0 Zip-First Contract

The top-level file still uses `items`; the difference is that each 2.0 item uses `release` instead of a folder `url`.

```json
{
  "schemaVersion": 3,
  "appName": "Example App",
  "items": [
    {
      "version": "2.0.0",
      "buildNumber": 200,
      "platform": "macos",
      "channel": "stable",
      "mandatory": false,
      "release": "https://updates.example.com/releases/example/2.0.0/macos/release.json"
    }
  ]
}
```

```json
{
  "schemaVersion": 3,
  "packageId": "com.example.app",
  "appName": "Example.app",
  "version": "2.0.0",
  "buildNumber": 200,
  "platform": "macos",
  "channel": "stable",
  "artifact": {
    "kind": "zip",
    "url": "https://cdn.example.com/releases/example/2.0.0/macos/Example-2.0.0-macos.zip",
    "sha256": "64-lowercase-hex-characters",
    "length": 12345678
  },
  "install": {
    "strategy": "wholeBundleReplace"
  },
  "signature": {
    "algorithm": "ed25519",
    "publicKeyId": "stable-2026-06",
    "value": "base64-signature-over-canonical-release-json-with-empty-signature-value"
  },
  "minimumUpdaterVersion": "2.0.0",
  "generatedAt": "2026-06-11T00:00:00Z"
}
```

The `signature` object is optional unless your application configures a production signature verifier. Length and SHA-256 verification are always part of the zip-first artifact check.

`buildNumber` is optional in 2.0 metadata. Keep it when your app exposes a
monotonic build number. Omit it when your app only exposes semantic versions;
2.0 then uses semantic version ordering and avoids treating a remote
`buildNumber` as newer than an equal installed version with no build metadata.

## Automated Migration CLI

Run the migrator from the root of the Flutter app that depends on
`desktop_updater`:

```sh
dart run desktop_updater:migrate --path .
```

The default mode is a dry run. It prints safe edits and manual findings without
writing files. After reviewing the report, apply safe edits with:

```sh
dart run desktop_updater:migrate --path . --apply
```

Use `--check` in CI when you want the command to fail if any safe edits or
manual findings remain:

```sh
dart run desktop_updater:migrate --path . --check
```

The migrator can safely update simple source changes:

- `pubspec.yaml` scalar dependency constraints such as
  `desktop_updater: ^1.3.0` to `desktop_updater: ^2.0.0`;
- `skipCheckVersion:` to `skipInitialVersionCheck:`;
- `getSkipCheckVersion` to `skipInitialVersionCheck`.

The migrator deliberately leaves these as manual findings:

- old `desktop_updater:release` and `desktop_updater:archive` commands, with
  the replacement `package` / `verify` flow in the report;
- custom typed-state UI migrations from boolean getters such as `needUpdate`,
  `isDownloading`, `isDownloaded`, and `downloadProgress`; stock
  `DesktopUpdateWidget`, `DesktopUpdateDirectCard`, `DesktopUpdateSliver`,
  `UpdateCard`, and `UpdateDialogListener` are available for 2.x state;
- low-level 1.x API usage such as `prepareUpdateApp`, `updateApp`, and
  `versionCheck`;
- nested, path, or git dependency declarations in `pubspec.yaml`;
- platform publishing changes, macOS signing/notarization/stapling, and
  Windows or Linux publisher-trust policy.

After applying safe edits, run:

```sh
flutter analyze
flutter test
dart run desktop_updater:verify --release path/to/release.json
```

The verify command should be run against the same descriptor and artifact bytes
that clients will download.

## Release Readiness Terms

2.0 separates two acceptance levels:

- `release-mechanics ready`: a Release build can check, download, verify,
  stage, install, relaunch, and roll back.
- `production-trusted ready`: the Release artifact also satisfies the
  platform's publisher-authenticity requirements.

Unsigned Windows and Linux Release builds can be accepted as
release-mechanics ready for many Flutter desktop apps. They should not be
described as production-trusted unless Authenticode, descriptor signatures, a
store channel, or another publisher-authenticity policy is in place.

macOS public direct distribution is different: production-trusted macOS updates
must be Developer ID signed, hardened-runtime enabled, notarized, stapled, and
Gatekeeper accepted. Apps that intentionally support unsigned macOS updates can
set `allowUnsignedMacOSUpdates: true`; that is an explicit owner opt-out and
keeps the macOS lane release-mechanics only.

## CLI Commands

Package a release:

```sh
dart run desktop_updater:package \
  --input build/macos/Build/Products/Release/Example.app \
  --output dist/2.0.0/macos \
  --package-id com.example.app \
  --app-name Example.app \
  --version 2.0.0 \
  --build-number 200 \
  --platform macos \
  --channel stable \
  --install-strategy wholeBundleReplace \
  --artifact-url https://cdn.example.com/releases/example/2.0.0/macos/Example.app.zip
```

Verify a release:

```sh
dart run desktop_updater:verify --release dist/2.0.0/macos/release.json
```

The verify command downloads the exact artifact URL from `release.json`, checks length and SHA-256, and validates zip extraction safety. Do not publish until this command passes against the same bytes clients will download.

`--build-number` is optional for `desktop_updater:package`. Omit it for
build-numberless apps.

## Typed State Migration

1.x code usually reads boolean controller fields:

```dart
if (controller.needUpdate && controller.isDownloaded) {
  await controller.restartApp();
}
```

2.0 code should prefer typed state:

```dart
switch (controller.state) {
  case UpdateReadyToInstall(:final stagingPath):
    await controller.restartApp();
  case UpdateFailed(:final error):
    debugPrint("Update failed: $error");
  default:
    break;
}
```

The 1.x compatibility getters are not present in the 2.x runtime API. Migrate
UI code to `controller.state` before upgrading shipped app code to 2.x.

## S3 And CDN Troubleshooting

- Do not require bucket listing.
- Publish exact URLs for `app-archive.json`, `release.json`, and the zip artifact.
- Signed URLs are supported when they remain valid long enough for the check and download flow.
- CDN URLs are supported when the client receives the exact descriptor and artifact bytes used to compute SHA-256.
- Avoid CDN transformations, automatic compression rewrites, or artifact re-zipping after `release.json` is generated.

## macOS Signing Troubleshooting

- Build a complete `.app` bundle, sign it, notarize it, staple it, and package it as a zip with `/usr/bin/ditto`.
- Keep bundle identifier and Team ID stable across releases.
- Ensure `codesign --verify`, Gatekeeper assessment, and stapler validation pass before publishing.
- A 2.0 macOS install replaces the whole `.app` bundle after native gates pass.
- Production macOS acceptance requires a Release app signed with a `Developer ID Application` identity, hardened runtime enabled, notarization completed, and the ticket stapled before the zip is published.
- The native helper rejects unsigned, unstapled, wrong-Team-ID, or wrong-bundle-ID staged apps in Release builds.
- Keep App Sandbox disabled for apps that use this whole-app replacement updater, and ensure production entitlements do not include `get-task-allow`. Sandboxed Mac App Store apps should not self-update with this plugin; use the App Store update channel instead.
- If you intentionally bypass macOS signing gates, pass `allowUnsignedMacOSUpdates: true` on `DesktopUpdaterController` or `DesktopUpdater.installUpdate`. The helper still requires a complete `.app` with the same `CFBundleIdentifier`, but it skips codesign, Gatekeeper, stapler, and Team ID validation. Do not call this production-trusted.

Notarization uses credentials saved in Keychain by profile name. Always keep the
exact `--keychain-profile` and `--keychain` pair printed by
`xcrun notarytool store-credentials`; use both values for `history`, `submit`,
and release automation:

```sh
xcrun notarytool store-credentials desktop-updater-notary \
  --key "$HOME/Developer/secrets/AuthKey_XXXXXXXXXX.p8" \
  --key-id "XXXXXXXXXX" \
  --issuer "00000000-0000-0000-0000-000000000000" \
  --keychain "$HOME/Library/Keychains/login.keychain-db" \
  --validate

xcrun notarytool history \
  --keychain-profile desktop-updater-notary \
  --keychain "$HOME/Library/Keychains/login.keychain-db" \
  --output-format json
```

If `store-credentials` succeeds but `history` or `submit` later reports
`No Keychain password item found`, the command is reading a different Keychain
than the one that stored the profile. Re-run `store-credentials` with explicit
`--keychain`, or pass the exact `--keychain` path shown in the success message.
Also check copied multi-line commands: the trailing `\` must be the final
character on the line, with no space after it.

Controller opt-out example:

```dart
final controller = DesktopUpdaterController(
  appArchiveUrl: Uri.parse("https://updates.example.com/app-archive.json"),
  allowUnsignedMacOSUpdates: true,
);
```

Manual install opt-out example:

```dart
await DesktopUpdater().installUpdate(
  stagingPath: "/path/to/staged/Example.app",
  allowUnsignedMacOSUpdates: true,
);
```

Run local production validation on the exported app:

```sh
codesign --verify --deep --strict --verbose=2 Example.app
spctl --assess --type execute --verbose=2 Example.app
xcrun stapler validate Example.app
codesign -dvvv --entitlements :- Example.app
```

Run the native production update smoke only with a prebuilt updated app that already contains the smoke sentinel before signing and notarization:

```sh
cd example
dart run tool/updater_smoke.dart \
  --app /Applications/Example.app \
  --production-gates \
  --staged-app dist/notarized/Example.app
```

## macOS SwiftPM And CocoaPods

2.0 treats Swift Package Manager as the primary macOS plugin integration path. CocoaPods remains supported as a fallback for apps that disable SwiftPM or still run older Flutter tooling.

Use SwiftPM for the normal lane:

```sh
flutter config --enable-swift-package-manager
cd example
flutter test integration_test -d macos
```

Use CocoaPods only for the fallback lane:

```sh
flutter config --no-enable-swift-package-manager
cd example
flutter test integration_test -d macos
```

Do not commit a `Podfile` in the SwiftPM-first example app. When SwiftPM is disabled, Flutter creates the CocoaPods files needed for the fallback lane and then runs `pod install`. A missing CocoaPods installation only blocks that fallback compatibility lane; it must not block SwiftPM validation.

## Windows Locked-File Troubleshooting

- The running `.exe` and loaded `.dll` files cannot be overwritten in-process.
- 2.0 schedules replacement with a native helper after the current process exits.
- The helper preserves the old app until replacement succeeds and rolls back on failure.
- Unsigned Release builds can be accepted as release-mechanics ready.
- Production-trusted direct distribution should Authenticode-sign `.exe` and `.dll` files and verify them with `signtool`.

## Linux Permissions Troubleshooting

- Publish a directory-bundle zip for 2.0 unless AppImage support is explicitly selected before final release.
- Do not rely on the current working directory.
- The helper resolves the running executable, validates the staging path, rejects removed paths outside the app root, and restores the previous app on failure.
- Unsigned Release builds can be accepted as release-mechanics ready.
- Direct zip distribution should add a release descriptor authenticity layer, such as Ed25519 descriptor signing or another pinned publisher signature policy, before it is treated as production-trusted.
